Syllabus: Safe Hybrid Development / Legacy Integration

Cyber Developer Associate (A-31) — Phase 7: Advanced Security Integration Module 70 hours CURRENT

Course Description

Course description pending.

Learning Outcomes

  1. Define what makes a system "legacy" — scale, age, tight coupling, and undocumented behavior — independent of its technology stack.
  2. Explain why a system that "still works" can carry latent security risk that integration activates.
  3. Describe the business pressure to integrate new functionality and the security cost of doing it unsafely.
  4. Articulate the course thesis: a legacy monolith must be understood and threat-modeled before any new service touches it. (OJL 2.c)
  5. Compare monolith and microservice architectures through a security lens — trust boundaries, blast radius, and attack surface.
  6. Define "hybrid development": new services operating alongside a living legacy core.
  7. Explain how a hybrid system expands the attack surface relative to either pure architecture. (OJL 2.c)
  8. Identify the new trust boundaries a hybrid introduces between legacy and new components.
  9. Define hidden dependencies — shared state, implicit contracts, and side effects — and explain why they're invisible during normal operation.
  10. Explain latent vulnerabilities that activate only under integration (the "booby trap" failure mode).
  11. Analyze real cases where integrating a new service introduced a new vulnerability through legacy behavior.
  12. Recognize and categorize the booby-trap classes an integrator must hunt before integrating. (OJL 2.c)
  13. Identify threat actors, trust boundaries, and attack vectors for a legacy monolith. (OJL 2.a)
  14. Analyze architecture and data-flow diagrams to surface vulnerabilities and risky assumptions. (OJL 2.a)
  15. Produce a pre-integration threat model that scopes where a new service can safely attach.
  16. Prioritize the threats that an integration would amplify. (OJL 2.c)
  17. Construct a systematic inventory of a legacy system's entry points, modules, data stores, and external integrations.
  18. Distinguish documented behavior from actual behavior and identify documentation gaps that hide integration risk.
  19. Apply safe, non-destructive discovery techniques that inventory a live legacy system without changing or breaking it.
  20. Produce a structured inventory artifact that feeds static and dynamic analysis later in the module. (OJL 2.c)
  21. Build a call graph and module-coupling map of a legacy monolith from its source, using the Lesson 5 inventory as the starting frame. (OJL 2.c)
  22. Identify shared mutable state and implicit contracts that are visible statically — the hidden dependencies most likely to become booby traps.
  23. Analyze a call graph as an architecture view to trace attack vectors and blast radius from a proposed integration seam. (OJL 2.a)
  24. Document static findings as candidate dependencies and risks, noting what only dynamic analysis can confirm.
  25. Apply tracing, logging, and instrumentation to observe a legacy system's actual runtime behavior and confirm the static map. (OJL 2.c)
  26. Detect side effects and hidden coupling — timing, ordering, and conditional behavior — that static analysis cannot see.
  27. Analyze observed runtime data flows as a refined architecture view to identify real attack paths and trust-boundary crossings. (OJL 2.a)
  28. Reconcile dynamic findings with the static map and resolve the requires-dynamic-confirmation items from Lesson 6.
  29. Record each discovered hidden dependency as a structured integration-risk entry with a clear trigger and impact.
  30. Score and prioritize each entry by likelihood of activation and security impact to produce an integration risk register.
  31. Determine which booby traps must be neutralized before integration versus monitored or accepted with justification. (OJL 2.c)
  32. Hand off the risk register as the bridge to Module 3 decomposition and Module 5 mitigation decisions.
  33. Apply the full discovery workflow — inventory, static analysis, dynamic analysis — to a provided legacy system end to end. (OJL 2.c)
  34. Produce a combined dependency and data-flow map and analyze it as an architecture view to identify attack vectors and a proposed integration seam. (OJL 2.a)
  35. Compile a scored integration risk register of the booby traps discovered, with neutralize/monitor/accept decisions.
  36. Present and defend the map and register, justifying findings against evidence and the threat model.
  37. Define a "seam" as a decomposition boundary and explain why seam choice is a security decision, not only an architectural one.
  38. Evaluate candidate seams against shared state and blast radius using the Module 2 dependency map and risk register.
  39. Sequence an integration so that the riskiest dependencies are isolated or neutralized before new code crosses the boundary. (OJL 2.c)
  40. Justify a chosen seam against the pre-integration threat model, naming the booby traps it contains and the residual risk it leaves.
  41. Explain the strangler fig pattern and how incremental replacement avoids the security risk of a big-bang rewrite.
  42. Identify the security controls required at the routing/facade layer that fronts the monolith during strangulation.
  43. Analyze the security considerations at each strangler step — dual-running, data ownership, and rollback — and how each can introduce or contain a booby trap. (OJL 2.c)
  44. Plan a secure strangulation sequence that leaves the hybrid system in a defensible state after every increment.
  45. Explain the anti-corruption layer (ACL) as a translation boundary that prevents the legacy model — and its hidden risk — from leaking into new services.
  46. Design the ACL's translation and validation responsibilities so every value crossing from legacy to new is sanitized and well-formed. (OJL 2.c)
  47. Treat the legacy side as untrusted: apply input validation, output encoding, and fail-closed defaults at the ACL to contain booby traps.
  48. Distinguish the ACL from an API gateway and decide where each belongs in a hybrid architecture.
  49. Explain how an API gateway centralizes authentication, rate limiting, and policy enforcement at the edge of a hybrid system.
  50. Apply the sidecar pattern to enforce cross-cutting security concerns consistently across legacy and new components. (OJL 2.c)
  51. Evaluate how gateways and sidecars contain legacy risk and where they themselves become new attack surface that must be hardened.
  52. Decide how gateway, sidecar, and anti-corruption layer compose into a coherent secure hybrid architecture.
  53. Select a safe seam for a provided legacy monolith using its Module 2 dependency map and integration risk register.
  54. Choose and combine the appropriate Module 3 patterns (strangler fig, anti-corruption layer, API gateway, sidecar) for the chosen boundary. (OJL 2.c)
  55. Produce a secure decomposition design that contains the register's booby traps and states residual risk.
  56. Justify the design against the pre-integration threat model and defend it under peer review.
  57. Explain why service-to-service authentication is mandatory at a hybrid seam, and why network-location trust ("it's inside the perimeter") is itself a booby trap.
  58. Compare mutual TLS and token-based (e.g. signed bearer/JWT) service identity, and select an approach appropriate to a legacy-to-new integration.
  59. Describe secure credential and key lifecycle management — issuance, rotation, revocation, and storage — for service identities.
  60. Evaluate an authentication design against a pre-integration baseline to confirm it adds the minimum necessary attack surface. (OJL 2.c)
  61. Distinguish authentication from authorization, and define a trust boundary as the line where one service stops trusting another by default.
  62. Apply least-privilege between services — scoping what a service may do, not just confirming who it is — and explain why implicit trust is a booby trap.
  63. Describe how to propagate and re-validate identity/authority across the seam (the confused-deputy problem) without granting the legacy core's privileges to the new service.
  64. Evaluate an authorization design against the pre-integration baseline to confirm it grants the minimum privilege needed. (OJL 2.c)
  65. Explain why an explicit API contract is a security control, and how implicit legacy contracts (Lesson 3) become booby traps at an integration seam.
  66. Define a strict contract — types, ranges, required/optional fields, formats, and a closed (default-deny) schema — for a cross-boundary interface.
  67. Implement schema validation on every cross-boundary payload, validating inbound and outbound, and rejecting rather than coercing malformed input.
  68. Version and evolve a contract without weakening its security guarantees, and confirm the contract adds no unnecessary surface against the baseline. (OJL 2.c)
  69. Protect data in transit across the seam — confidentiality, integrity, and authenticity — and explain what shared data stores expose at a hybrid boundary.
  70. Analyze consistency patterns (synchronous vs. eventual, shared database vs. message-based) for the security risks each introduces between legacy and new services.
  71. Identify how consistency mechanisms can re-open race conditions and injection paths (shared-state and timing booby traps from Module 1) and apply safeguards.
  72. Evaluate a data-exchange design against the pre-integration baseline to confirm it minimizes new attack surface. (OJL 2.c)
  73. Define attack surface for a hybrid system and explain how integration expands it across endpoints, ports, privileges, credentials, and data paths.
  74. Establish a pre-integration surface baseline and compute the surface delta an integration introduces.
  75. Apply surface-reduction techniques — closing unnecessary endpoints/ports, dropping privileges, and removing redundant or legacy-fallback paths.
  76. Review a completed integration's post-integration surface against the baseline and justify every retained addition. (OJL 2.c)
  77. Apply the "treat both sides as untrusted" principle, validating data crossing the seam in both directions between legacy and new services.
  78. Implement layered input validation at the boundary — type, range, format, and semantic checks — using allow-list rather than deny-list strategies.
  79. Apply correct output encoding and context-aware sanitization so data passed across the seam cannot weaponize a latent legacy flaw.
  80. Design fail-closed error handling at the boundary that neither leaks information nor leaves the seam in an exploitable partial state. (OJL 2.c)
  81. Identify how secrets and configuration diverge between a legacy monolith and new services, and the new exposure the hybrid seam creates.
  82. Apply secure secret-management practices — externalized storage, least-privilege access, and rotation — across both legacy and new sides of the integration.
  83. Prevent secret leakage through logs, error messages, configuration files, and version control at the boundary. (OJL 2.c)
  84. Manage configuration as a security-relevant surface, using fail-safe defaults and environment separation across the hybrid system.
  85. Establish a pre-integration security baseline against which the post-integration attack surface can be compared to prove no new exposure.
  86. Design security regression tests that re-run on every change and fail the build when a known-good security property regresses.
  87. Author abuse-case and edge-case tests at the seam that attempt to trigger the booby-trap classes discovered in earlier modules. (OJL 2.c)
  88. Interpret and act on test results — distinguishing a real regression from noise, and gating release on a proven-clean surface.
  89. Explain why testing is necessary but not sufficient, and the role of production monitoring as the last line of defense against an activated booby trap.
  90. Instrument the seam with observability — structured logs, metrics, and traces — that surfaces hidden-dependency activation between legacy and new services.
  91. Define alerts on booby-trap trigger signals and tune them to be actionable rather than noisy. (OJL 2.c)
  92. Design safe, fast rollback and containment so a fired booby trap is reverted before it becomes an incident.
  93. Implement a new service's integration against a provided legacy monolith using the Module 4 design and Module 5 secure-coding practices at the seam.
  94. Apply cross-boundary validation, secure secret/config handling, and fail-closed error handling so the seam treats both sides as untrusted.
  95. Run the security regression and abuse-case suite against the integration and confirm a clean, justified surface delta. (OJL 2.c)
  96. Document the result — surface delta, passing trap tests, and a detection/rollback plan — as evidence the integration introduced no new vulnerability.
  97. Read and summarize a provided legacy monolith and its new-service integration goal, identifying the business driver and the integration seam in scope.
  98. Produce a pre-integration threat model of the monolith, naming threat actors, trust boundaries, and likely booby-trap classes to hunt.
  99. Draft a capstone integration plan that sequences discovery, design, implementation, and validation across the remaining lab lessons. (OJL 2.c)
  100. Define the "done" bar for the capstone: a working integration plus evidence it introduced no new vulnerability. (OJL 2.c)
  101. Apply static and dynamic analysis to the capstone monolith to surface hidden dependencies along the integration seam.
  102. Record each discovered hidden dependency as a scored entry in an integration risk register, classified by booby-trap type.
  103. Decide and apply a neutralization action for each high-priority booby trap — mitigate, isolate, or contain — before any integration code is written. (OJL 2.c)
  104. Confirm the seam is safe to build against, producing a discovery summary that hands off to the implementation stage. (OJL 2.c)
  105. Apply a chosen secure architecture pattern (strangler fig, anti-corruption layer, gateway, or sidecar) to create a safe seam for the new service.
  106. Implement secure service-to-service integration — authentication, authorization, and a validated API contract — across the seam.
  107. Code the seam defensively, honoring the Lesson 26 design constraints so no neutralized booby trap is re-armed and no new attack surface is opened. (OJL 2.c)
  108. Produce a working integration plus a post-integration surface baseline ready for validation. (OJL 2.c)
  109. Run security regression and abuse-case tests against the integrated system to prove no new vulnerability was introduced at the seam.
  110. Produce a surface delta comparing the pre-integration baseline to the post-integration surface, accounting for every change. (OJL 2.c)
  111. Document residual risk and the production monitoring needed to detect any deferred booby-trap trigger. (OJL 2.c)
  112. Assemble the capstone evidence package against the Lesson 25 "done" bar and reflect on the end-to-end safe-integration process. (OJL 2.c)

Course Outline

1 Legacy Systems & the Hybrid Security Challenge 10 hours
Lesson: 1: What Makes a System "Legacy" — and Why It's Risky (2h)
  • Characteristics of legacy monoliths: scale, age, undocumented behavior, tight coupling
  • Why "it still works" hides security risk
  • Business pressure to integrate vs. the cost of doing it unsafely
Lesson: 2: Monoliths, Microservices, and the Hybrid Middle Ground (2h)
  • Monolith vs. microservice trade-offs from a security lens
  • What "hybrid development" means: new services alongside a living legacy core
  • The expanded attack surface a hybrid system creates
Lesson: 3: The "Booby Trap" Problem — Hidden Dependencies & Latent Vulnerabilities (3h)
  • Hidden dependencies: shared state, implicit contracts, side effects
  • Latent vulnerabilities that activate only under integration
  • Case studies of integrations that introduced new vulnerabilities
Lesson: 4: Threat-Modeling a Legacy Monolith (3h)
  • Identifying threat actors, trust boundaries, and attack vectors (supports OJL 2.a)
  • Data-flow and architecture-diagram analysis of a monolith
  • Producing a pre-integration threat model
2 Mapping the Monolith — Discovering Hidden Dependencies 12 hours
Lesson: 5: Inventorying a Legacy Codebase (2h)
  • Establishing what exists: entry points, modules, data stores, external integrations
  • Documentation gaps and how to fill them safely
Lesson: 6: Static Dependency Analysis — Call Graphs & Coupling (3h)
  • Building call graphs and module-coupling maps
  • Spotting shared mutable state and implicit contracts statically
Lesson: 7: Dynamic Analysis — Observing Runtime Behavior (3h)
  • Tracing, logging, and instrumentation to reveal runtime dependencies
  • Detecting side effects and hidden coupling that static analysis misses
Lesson: 8: Cataloging Booby Traps — Risk Register & Integration Risk Assessment (2h)
  • Recording each hidden dependency as a scored integration risk
  • Prioritizing what must be neutralized before integration
Lesson: 9: Hands-On — Map a Monolith's Attack & Dependency Surface (2h)
  • Apply static + dynamic analysis to a provided legacy system
  • Produce a dependency map and integration risk register
3 Secure Hybrid Architecture Patterns 12 hours
Lesson: 10: Decomposition Strategy — Choosing Safe Seams (2h)
  • Where to cut: boundaries that minimize shared state and blast radius
  • Sequencing integration to reduce risk
Lesson: 11: The Strangler Fig Pattern, Securely (3h)
  • Incrementally replacing legacy functionality without big-bang risk
  • Security considerations at each strangler step
Lesson: 12: Anti-Corruption Layer — Containing Legacy Risk (3h)
  • Translating between legacy and new models to prevent risk leakage
  • Validating and sanitizing at the boundary
Lesson: 13: API Gateways & Sidecars — Securing the Edge of the Hybrid (2h)
  • Centralizing auth, rate limiting, and policy at the gateway
  • Sidecar patterns for secure cross-cutting concerns
Lesson: 14: Hands-On — Design a Secure Decomposition Boundary (2h)
  • Given the Module 2 risk register, design a safe seam + pattern choice
  • Justify the boundary against the threat model
4 Secure Service Integration 14 hours
Lesson: 15: Service-to-Service Authentication (mTLS, Tokens) (3h)
  • Mutual TLS and token-based service identity
  • Key/secret management for service credentials
Lesson: 16: Authorization & Trust Boundaries Across Services (2h)
  • Least-privilege between services; avoiding implicit trust
  • Propagating and validating identity across the seam
Lesson: 17: Secure API Contracts & Schema Validation (3h)
  • Defining strict contracts; validating every cross-boundary payload
  • Versioning contracts without weakening security
Lesson: 18: Secure Data Exchange & Consistency Across the Seam (3h)
  • Protecting data in transit between legacy and new services
  • Consistency patterns that don't open injection or race vulnerabilities
Lesson: 19: Minimizing New Attack Surface During Integration (3h)
  • Closing unnecessary endpoints, ports, and privileges introduced by integration
  • Reviewing the post-integration surface against the pre-integration baseline
5 Integrating Without Introducing Vulnerabilities 12 hours
Lesson: 20: Secure Coding at the Seams — Input Validation Across Boundaries (3h)
  • Treating the legacy side as untrusted (and vice versa)
  • Cross-boundary validation, encoding, and error handling
Lesson: 21: Secrets & Configuration in a Hybrid System (2h)
  • Managing secrets/config across legacy and new services
  • Avoiding leakage through logs, configs, and error messages
Lesson: 22: Security Regression Testing the Integration (3h)
  • Building tests that prove no new vulnerability was introduced
  • Edge-case and abuse-case testing at the boundary
Lesson: 23: Monitoring & Detecting Booby-Trap Triggers in Production (2h)
  • Observability for the seam: detecting activated hidden dependencies
  • Alerting and safe rollback when a booby trap fires
Lesson: 24: Hands-On — Integrate a Service Without New Vulnerabilities (2h)
  • Integrate a new service against the Module 4 design
  • Run security regression checks; confirm a clean surface
6 Capstone Integration Lab — Safe Hybrid Build 10 hours
Lesson: 25: Scenario & Planning — The Legacy System & Integration Goal (2h)
  • Understand a provided legacy monolith and the new-service goal
  • Threat-model and plan the integration
Lesson: 26: Discover & Neutralize Booby Traps (3h)
  • Map dependencies; identify and mitigate hidden risks before integration
Lesson: 27: Implement the Secure Integration (3h)
  • Apply a secure pattern + secure service integration end to end
Lesson: 28: Validate No New Vulnerabilities & Document (2h)
  • Security regression test the result; document the surface delta and residual risk
