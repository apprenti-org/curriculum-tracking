Learning Outcomes

  1. Trace an HTTP request through the Spring MVC pipeline — from `DispatcherServlet` to `HandlerMapping` to the matched controller method — and explain why this lifecycle is the map of an endpoint's attack surface.
  2. Explain how HTTP request bodies and JSON deserialize and bind into controller method parameters and DTOs through Jackson and Spring's argument resolvers.
  3. Identify every binding annotation — `@RequestBody`, `@RequestParam`, `@PathVariable`, `@RequestHeader`, `@CookieValue` — as a distinct channel through which untrusted data enters the application.
  4. Map the complete attack surface of a single Spring endpoint by enumerating every untrusted source that reaches the controller, and explain why that map is the unit of analysis for every defense in this course.
  5. Construct a Spring `@RestController` with explicit request mappings and bind a JSON request body to a typed DTO with `@RequestBody`, naming the exact point where untrusted input crosses into the application.
  6. Explain why the controller boundary is the application's first input-trust boundary and why a typed DTO is a stronger contract than reading raw maps or loose parameters.
  7. Enforce the shape and bounds of a request DTO using Jakarta Bean Validation constraint annotations (`@NotNull`, `@NotBlank`, `@Size`, `@Min`/`@Max`, `@Pattern`, `@Email`) and trigger them at the boundary with `@Valid`.
  8. Reject malformed or hostile requests before they reach business logic by relying on `MethodArgumentNotValidException` and a fail-closed validation posture, choosing allow-list constraints over blocklist filtering.
  9. Explain why an HTTP API — not just a browser-rendered page — benefits from security response headers, and identify the headers that matter at an API boundary and the threat each one blunts.
  10. Configure security response headers on Spring endpoints through `HttpSecurity.headers()`, setting `Content-Security-Policy`, `X-Content-Type-Options`, `Strict-Transport-Security`, and related headers deliberately rather than relying on defaults.
  11. Explain what Cross-Site Request Forgery (CSRF) is, why it depends on ambient credentials, and reason through whether a given REST endpoint needs CSRF protection enabled or can safely disable it.
  12. Apply Spring's `CsrfConfigurer` correctly for the endpoint's authentication model — keeping CSRF protection on for cookie/session-authenticated endpoints and disabling it deliberately (with justification) for stateless token-authenticated APIs.
  13. Configure Cross-Origin Resource Sharing (CORS) as a least-privilege allow-list using `CorsConfigurationSource`, enumerating allowed origins, methods, and headers, and explain why a `` wildcard — especially combined with credentials — is dangerous.
  14. Explain where the Spring Security filter chain sits in the request lifecycle — in front of the `DispatcherServlet` — and why placing security at the servlet-filter layer means a request is screened before any controller code runs.
  15. Configure a `SecurityFilterChain` bean using component-based configuration, and explain why the deprecated `WebSecurityConfigurerAdapter` is no longer the pattern in Spring Security 6.x.
  16. Distinguish authentication (establishing who is calling) from authorization (deciding what that caller may do), and trace how the `AuthenticationManager`, the `Authentication` token, and the `SecurityContext` carry identity through a request.
  17. Protect endpoints by role and authority with `authorizeHttpRequests`, applying a deny-by-default, least-privilege posture so that every path is locked down unless explicitly opened.
  18. Apply method-level authorization with `@PreAuthorize` to enforce fine-grained, least-privilege checks at the service boundary as a defense-in-depth layer behind the URL rules.
  19. Explain how Spring's default error responses leak stack traces, exception class names, and other system internals, and why that information disclosure aids an attacker fingerprinting the application.
  20. Implement centralized exception handling with `@ControllerAdvice` and `@ExceptionHandler` (or `ResponseEntityExceptionHandler`) so that every failure returns a controlled, safe response instead of a default leak.
  21. Return useful, non-revealing error payloads — a structured `ProblemDetail` (RFC 7807) body mapped to the correct HTTP status code — while logging the full exception detail server-side only.

Course Outline

1 Spring MVC Foundations 3 hours
Lesson: 1: HTTP, JSON, and the Spring MVC Request Lifecycle (1.5h)
  • The request lifecycle: from `DispatcherServlet` through handler mapping to the controller
  • How HTTP bodies and JSON bind to method parameters and DTOs
  • Where untrusted input enters — the attack surface of a Spring endpoint
Lesson: 2: Controllers, Request Mapping, and Input Validation (1.5h)
  • `@RestController`, request mapping, and DTO-based request binding
  • Bean Validation (`@Valid`, constraint annotations) to enforce shape and bounds
  • Rejecting malformed or hostile requests before they reach business logic
2 Securing the Endpoints 4 hours
Lesson: 3: Secure REST Endpoint Design — Headers, CSRF, and CORS (1.5h)
  • Security response headers and why they matter for an API
  • CSRF protection: when it applies to REST endpoints and how Spring enforces it
  • Configuring CORS as a least-privilege allow-list, not a wildcard
Lesson: 4: Spring Security Essentials — Authentication and Authorization (1.5h)
  • The Spring Security filter chain and where it sits in the request lifecycle
  • Authentication primitives: establishing who is calling
  • Authorization primitives: protecting endpoints by role and least privilege
Lesson: 5: Secure Error Handling — No Stack-Trace Leakage (1h)
  • How default error responses leak stack traces, types, and system internals
  • Centralized exception handling (`@ControllerAdvice` / `@ExceptionHandler`) for safe responses
  • Returning useful, non-revealing error payloads and correct status codes
3 Secure-Endpoint Capstone 3 hours
Lesson: 6: Secure-Endpoint Capstone Lab (3h)
  • Build a REST endpoint that validates input, authenticates and authorizes callers, sets correct headers/CSRF/CORS, and handles errors safely
  • Audit the endpoint against the course's secure-coding controls
  • Demonstrate the endpoint resists a scripted set of attacks (bad input, unauthorized access, error-leak probes)
