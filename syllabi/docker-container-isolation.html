Cyber Developer Associate — Phase 4 (Secure Data Management) 16 hours CURRENT

Learning Outcomes

  1. Explain what a container is and how it differs from a virtual machine in terms of what each isolates.
  2. Describe what container isolation buys you for security — blast-radius containment, reproducibility, and least exposure.
  3. State the course's central problem: standing up a secure, isolated database environment.
  4. Describe how namespaces give a container its isolated filesystem, network, and process view.
  5. Explain how control groups (cgroups) bound a container's resource consumption.
  6. Identify what is isolated by default and what is not, and how the host mediates the boundary.
  7. Explain what a container image is and how tags and digests identify it.
  8. Describe what a registry is and how image provenance establishes whether a base image is trustworthy.
  9. Classify base-image choices as supply-chain security decisions.
  10. Apply the core container lifecycle commands — `run`, `create`, `start`, `stop`, `ps`, and `rm` — to manage isolated environments.
  11. Apply `inspect` and an in-container view to observe the isolation boundary directly.
  12. Identify container state and read it when something looks missing.
  13. Pull a trusted database image from a registry and run it as an isolated container.
  14. Distinguish ephemeral from persistent database containers and reason about when each is appropriate.
  15. Publish only the ports a database environment genuinely needs, treating every published port as attack surface.
  16. Author a Dockerfile using the core instructions — `FROM`, `WORKDIR`, `COPY`, `RUN`, `CMD` — and build and tag the resulting image.
  17. Keep an image minimal and purpose-scoped, preferring `COPY` over `ADD` and reasoning about why fewer instructions mean less to exploit.
  18. Build a database image scoped to exactly one job, so the image carries only what that job requires.
  19. Distinguish named volumes from bind mounts and reason about the data lifecycle as independent of the container.
  20. Protect database data at rest in isolated storage, controlling ownership and access to the persisted data.
  21. Back up and restore a volume-backed database so persisted data survives container rebuilds.
  22. Distinguish a user-defined Docker network from the default bridge and explain how each affects reachability.
  23. Restrict reachability so only the application that needs the database can reach it, using a private network rather than a published port.
  24. Stand up an app-and-database pair where the database port is never exposed to the host or the world by default.
  25. Run a container as a non-root user and explain why root inside a container is a privilege you almost never need.
  26. Drop unneeded Linux capabilities so a container holds only the privileges its job requires.
  27. Apply a read-only root filesystem and `no-new-privileges` so an escaped or compromised container lands somewhere harmless.
  28. Explain why a credential placed in an image or any of its layers is exposed permanently, even if a later step appears to remove it.
  29. Inject database credentials at run time so they never become part of the image.
  30. Apply Docker secrets and environment-injection patterns to supply credentials to a database environment safely.
  31. Detect a credential that has leaked into a build and remediate it correctly.
  32. Choose a minimal or distroless base image and explain how a smaller image shrinks the attack surface.
  33. Scan an image for known vulnerabilities and interpret the findings.
  34. Establish a patch and rebuild cadence that keeps a database image current against newly disclosed vulnerabilities.
  35. Maintain an isolated database environment over time through rebuilds and image refreshes, recognizing and correcting drift.
  36. Apply log hygiene and inspect a long-running environment to confirm the isolation boundary still holds.
  37. Safely decommission an isolated environment, removing containers, volumes, and secrets without leaving sensitive remnants.

Course Outline

1 Container Isolation Fundamentals 4 hours
Lesson: 1: Why Containers — Isolation as a Security Boundary (1h)
  • Virtual machines versus containers, and what "isolation" actually means
  • What isolation buys you for security: blast-radius containment, reproducibility, least exposure
  • The course's central problem: secure, isolated database environments
  • Activity type: demonstration
Lesson: 2: The Anatomy of Container Isolation (1h)
  • Namespaces, control groups (cgroups), and the isolated filesystem/network/process view
  • What is isolated by default — and what is not
  • How the host mediates the boundary the container sees
  • Activity type: demonstration
Lesson: 3: Images, Registries, and Trusted Base Images (1h)
  • Images as templates; tags and image digests
  • Registries and image provenance; choosing a trusted base image
  • Base-image selection as a supply-chain security decision
  • Activity type: exercise
Lesson: 4: Core Docker CLI for Isolated Environments (1h)
  • `run` / `create` / `start` / `stop` / `ps` / `inspect` / `rm`
  • Observing the isolation boundary directly (processes, mounts, network from inside vs. outside)
  • Reading container state when something looks missing
  • Activity type: code-along
2 Building & Running Isolated Database Environments 5 hours
Lesson: 5: Running a Database in a Container (1h)
  • Pulling and running a database image
  • Ephemeral versus persistent containers
  • Port publishing as attack surface — expose only what is needed
  • Activity type: code-along
Lesson: 6: Authoring a Dockerfile for a Hardened DB Image (1.5h)
  • `FROM` / `COPY` / `RUN` / `CMD` / `WORKDIR` and the build/tag lifecycle
  • Minimal, purpose-scoped layers; `COPY` over `ADD`; why fewer instructions means less to exploit
  • Building a database image scoped to exactly one job
  • Activity types: code-along, exercise
Lesson: 7: Persisting Data Securely with Volumes (1h)
  • Named volumes versus bind mounts; the data lifecycle independent of the container
  • Protecting data at rest in isolated storage; backups and ownership
  • Activity type: code-along
Lesson: 8: Network Isolation for Database Environments (1.5h)
  • User-defined networks versus the default bridge
  • Restricting reachability so only the application that needs the database can reach it
  • Never exposing the database port to the host/world by default
  • Activity types: code-along, exercise
3 Hardening & Securing Information Through Isolation 5 hours
Lesson: 9: Least-Privilege Containers — Non-Root and Capabilities (1h)
  • Running as a non-root user; dropping Linux capabilities
  • Read-only root filesystems and `no-new-privileges`
  • Why a container that escapes should land somewhere harmless
  • Activity types: code-along, exercise
Lesson: 10: Secrets Management — Keeping Credentials Out of Images (1.5h)
  • Why credentials must never live in an image or its layers (layer history is forever)
  • Injecting DB credentials at run time; Docker secrets and environment-injection patterns
  • Detecting a credential that leaked into a build
  • Activity types: code-along, exercise
Lesson: 11: Hardening Images — Minimal Base, Scanning, and Updates (1.5h)
  • Minimal and distroless base images to shrink attack surface
  • Image scanning for known vulnerabilities
  • A patch/rebuild cadence that keeps an image current
  • Activity types: code-along, exercise
Lesson: 12: Maintaining Isolated Environments Over Time (1h)
  • The "maintain" half of OJL 5.a: rebuilds, image refreshes, and drift
  • Log hygiene and inspecting a long-running environment
  • Safe decommissioning — removing containers, volumes, and secrets
  • Activity type: exercise
4 Capstone 2 hours
Lesson: 13: Capstone Lab — Secure Isolated Database Environment (2h)
  • Assemble a complete isolated database environment: a non-root, network-isolated, volume-backed database container with credentials kept out of the image
  • Demonstrate the isolation boundary holds — no unintended reachability, no secrets in the image, least privilege in effect
  • Defend the isolation and hardening choices made
  • Activity type: hands-on capstone lab (adapted capstone structure per `capstone-design-process.md`)
