Learning Outcomes

  1. Explain why the Model Context Protocol exists and how its host, client, and server roles route context to a language model.
  2. Explain why an MCP server is a security boundary rather than a passive connector.
  3. Frame the course's central problem: providing useful local context to a model without exposing the underlying database.
  4. Distinguish the three MCP primitives — tools, resources, and prompts — by what each one does.
  5. Choose the appropriate primitive for a given context-provision need.
  6. Identify the data-exposure surface each primitive creates and its security implications.
  7. Set up the official Python MCP SDK and scaffold a runnable server.
  8. Define a single read-only tool that returns a scoped answer from a local transaction store.
  9. Run the server over the stdio transport.
  10. Connect a client and trace a request and response across the boundary.
  11. Apply the least-privilege principle to scope what a model can see through the server.
  12. Choose between resources and tools for exposing stored data based on exposure risk.
  13. Return safe, shaped views of data instead of raw rows.
  14. Recognize and correct common over-exposure anti-patterns.
  15. Apply scoped retrieval so the server reads only the data a task requires.
  16. Parameterize all data access at the boundary to prevent injection into queries.
  17. Keep raw SQL and database connections entirely off the model side.
  18. Validate tool arguments against a strict schema before they are used.
  19. Sanitize data returned across the boundary to prevent sensitive-data leakage.
  20. Recognize and defend against prompt-injection vectors carried in data.
  21. Distinguish authentication from authorization in the context of an MCP server.
  22. Authorize access per operation so each tool exposes only what its caller may use.
  23. Configure least-privilege credentials for the server's own access to the data store.
  24. Return errors across the boundary that reveal no schema, stack traces, or sensitive values.
  25. Log server activity usefully without recording secrets or personally identifiable information.
  26. Design failure modes that keep the boundary intact under error conditions.

Course Outline

1 MCP Foundations & Your First Secure Server 5 hours
Lesson: 1: What MCP Is and the Trust Model (1h)
  • Why the Model Context Protocol exists: standardized context for language models
  • The host / client / server architecture and how messages flow
  • The MCP server as a security boundary, not just a connector
  • Local context versus direct database access — framing the course's central problem
  • Activity type: demonstration
Lesson: 2: MCP Primitives — Tools, Resources, and Prompts (1h)
  • Tools (model-invoked actions), resources (readable data), and prompts (reusable templates)
  • Choosing the right primitive for a given need
  • The data-exposure surface each primitive creates
  • Activity type: exercise
Lesson: 3: Building Your First Secure Server with the Python SDK (1.5h)
  • Setting up the official Python MCP SDK
  • Defining a single read-only tool over a local transaction store
  • Serving over the stdio transport
  • Connecting a client and observing the boundary in action
  • Activity type: code-along
Lesson: 4: Exposing Data with Least Privilege — Resources vs Tools (1.5h)
  • Scoping what the model can see to the minimum required
  • Resources versus tools for exposing stored data
  • Returning safe, shaped views instead of raw rows
  • Activity types: code-along, exercise
2 Hardening the Boundary 5 hours
Lesson: 5: Preventing Database Exposure at the Boundary (1h)
  • Scoped queries instead of open data access
  • Parameterization at the boundary
  • Never handing the model raw SQL or a database connection
  • Activity types: code-along, exercise
Lesson: 6: Validating Inputs and Sanitizing Returned Data (1h)
  • Validating tool arguments before they reach data access
  • Sanitizing data returned across the boundary
  • Prompt-injection and sensitive-data-leakage vectors and defenses
  • Activity types: code-along, exercise
Lesson: 7: Authentication and Authorization for MCP Servers (1h)
  • Authenticating callers of an MCP server
  • Authorizing access per tool and per resource
  • Least-privilege credentials for the server's own data access
  • Activity types: code-along, exercise
Lesson: 8: Secure Error Handling and Logging (1h)
  • Error handling that leaks no schema, stack traces, or sensitive values
  • Logging that records what is needed without exposing secrets or personally identifiable information
  • Failure modes that keep the boundary intact
  • Activity types: code-along, exercise
Lesson: 9: Capstone Lab — Shielded Transaction Context (1h)
  • Assemble a secure MCP server that provides transaction context to a language model with the database fully shielded
  • Demonstrate least-privilege exposure, input validation, sanitization, authorization, and safe error handling together
  • Verify the boundary holds against database-exposure and injection attempts
  • Activity type: hands-on capstone lab (adapted capstone structure per `capstone-design-process.md`)
