Learning Outcomes

  1. Provision a small distributed service to a cloud sandbox (or local cloud emulation) using declarative Infrastructure-as-Code — choosing tooling, file layout, and module boundaries appropriate to the service.
  2. Configure at least one network-layer concern (DNS, firewall, load balancer, or routing) as part of the IaC and document why that choice fits the service.
  3. Apply least-privilege access controls to at least one resource in the IaC, with a documented secrets-management approach for any credentials the service requires.
  4. Demonstrate reproducible build from clean state — destroying and re-creating the environment from IaC at least once — and commit the infrastructure with intent-explaining commit messages.
  5. Configure at least one observability signal (metrics, logs, or traces) on the stood-up service, with sufficient data to reconstruct service behavior after the fact.
  6. Define at least one alert rule that triggers on an operationally meaningful condition, and verify alert delivery via end-to-end test.
  7. Declare explicit SLA / SLO / OLA targets for the service before the incident phase, with defensible rationale.
  8. Author a runbook covering deployment procedure, monitoring access paths, common operations, rollback steps, and known limitations — readable by an operator unfamiliar with this specific service.
  9. Receive an operational alert and use the alert payload + runbook to orient quickly.
  10. Investigate alert context from monitoring data to scope blast radius before any remediation.
  11. Make and document an explicit severity classification with named rationale tied to operational impact.
  12. File a complete initial incident triage ticket: timestamp, signal source, severity, blast radius, working hypothesis.
  13. Cross-reference monitoring data across signal classes (metrics + logs + traces) to narrow a suspected cause class.
  14. Identify which stakeholders require notification at the chosen severity level and trigger those notifications per the runbook.
  15. Update the triage ticket with a refined working hypothesis and the next-step investigation plan.
  16. Distinguish working hypothesis (provisional) from attribution (claim about root cause) in incident communications.
  17. Open and run an incident bridge using a documented escalation pattern with severity-appropriate cadence.
  18. Direct investigative experiments against the working hypothesis, recording each experiment's pre-state, execution, observed outcome, and what it tells you.
  19. Coordinate working theories with monitoring evidence in real time without over-attributing prematurely.
  20. Record decision points with timestamps such that the timeline can be reconstructed accurately afterward.
  21. Apply a restoration intervention with documented blast-radius assumption and rollback plan, under time pressure.
  22. Validate restoration against monitoring signals — confirming the SLA / SLO impact window has closed by data, not by assumption.
  23. Document remaining risk and watch-list items for the post-restoration window.
  24. Recognize fault-tolerance gaps surfaced by the incident and capture them for Module 4's automation improvement.
  25. Compile a chronological incident timeline from the bridge + restoration logs, distinguishing significant events from observations.
  26. Distinguish apprentice-internal events from stakeholder-external events when summarizing.
  27. Cross-reference monitoring data and decisions in the timeline so the story is anchored in evidence, not memory.
  28. Annotate the Lesson-2 runbook with gaps surfaced during the incident.
  29. Establish the incident timeline (from Lesson 7 source data) framed for retrospective analysis rather than chronological narration.
  30. Distinguish symptoms (what failed visibly) from causes (what produced the failure) in writing.
  31. Identify the proximate trigger and one or more contributing causes.
  32. Propose preventive changes tied to each contributing cause, scoped specifically enough for Lesson 9 implementation.
  33. Document SLA / SLO / OLA impact: budget consumed, business consequence, residual budget.
  34. Select one preventive change from the RCA using documented decision criteria.
  35. Design an automation: trigger, behavior, idempotency, rollback path, observability.
  36. Conduct an impact assessment: blast radius, rollback strategy, stakeholder plan, rollout sequencing.
  37. Produce a peer-reviewable design document.
  38. Implement the automation designed in Lesson 9, in whichever form the design requires.
  39. Make the automation idempotent and reproducible — re-runnable without producing duplicate side effects.
  40. Update the runbook (Lesson 2) to reflect the new automation: invocation, expected behavior, failure modes, rollback.
  41. Validate against monitoring post-deploy, confirming observability hooks emit expected signals.
  42. Produce a cross-functional handoff artifact appropriate to a non-technical stakeholder audience.
  43. Demonstrate the incident resolution and automation improvement in a form a non-technical reviewer can verify.
  44. Compile the apprentice-authored evidence map per §9 of `capstone-design-process.md`.
  45. Participate in the CBR signoff transaction, including per-competency revision for any Beginning / Developing ratings.

Course Outline

1 Stand Up the Monitored Service 8 hours
Lesson: 1: IaC Stand-Up (4h)
  • Provision compute, network, and storage via declarative IaC (Terraform or equivalent)
  • Apply least-privilege access controls and secrets-management approach
  • Configure at least one network-layer concern (DNS, firewall rule, load balancer, or routing) and at least one cloud resource
  • Commit infrastructure to version control with intent-explaining commit messages
Lesson: 2: Monitoring + Runbook (4h)
  • Configure metrics, logs, or traces for the service
  • Define at least one alert rule on operationally meaningful conditions; verify end-to-end alert delivery
  • Author the runbook: deployment procedure, monitoring access paths, rollback steps
  • Declare SLA / SLO / OLA targets before the incident phase begins
2 Detect and Triage 6 hours
Lesson: 3: Alert Ingestion + Severity Call (3h)
  • Receive the scripted alert from monitoring
  • Investigate from monitoring data to scope blast radius and identify affected systems
  • Classify severity with recorded rationale
  • File the initial incident ticket
Lesson: 4: Triage Investigation (3h)
  • Use monitoring + logs + traces to narrow the suspected cause class
  • Identify stakeholders requiring notification at the chosen severity level
  • Hand off to the bridge phase with the ticket updated
3 Run the Incident 10 hours
Lesson: 5: Incident Bridge Coordination (4h)
  • Run the simulated bridge (live or async with simulated stakeholders)
  • Follow the escalation pattern appropriate to severity (status cadence, stakeholder communications)
  • Record decision points with timestamps
Lesson: 6: Service Restoration (3h)
  • Apply the restoration intervention(s)
  • Validate restoration against monitoring signals; confirm SLA / SLO impact window
  • Document remaining risk and watch-list items
Lesson: 7: Communications Timeline (3h)
  • Compile the bridge communications into a coherent chronological record
  • Distinguish learner-visible events from supervisor-only events
  • Cross-reference monitoring data + decisions in the timeline
4 Root Cause Analysis + Automate 10 hours
Lesson: 8: Root Cause Analysis Document (4h)
  • Establish the incident timeline from monitoring and event sources
  • Distinguish symptoms from causes in writing
  • Identify the proximate trigger and at least one contributing cause
  • Propose preventive changes tied to each contributing cause
  • Document SLA / SLO impact in the RCA
Lesson: 9: Automation Design + Impact Assessment (3h)
  • Choose one preventive change to automate (script, IaC change, monitoring config, runbook revision)
  • Design the automation: trigger, behavior, idempotency, rollback path
  • Perform an impact assessment: blast radius, rollback strategy, stakeholder notification plan
Lesson: 10: Automation Implementation (3h)
  • Build and merge the automation improvement
  • Demonstrate reproducible / re-runnable execution from clean state
  • Update the runbook to reflect the new automation
  • Validate against monitoring after deploy
5 Cross-Functional Handoff + Capstone Signoff 6 hours
Lesson: 11: Handoff Artifact + CBR Signoff (6h)
  • Produce the cross-functional handoff artifact: doc, deck, or recorded walkthrough appropriate to non-technical stakeholders
  • Demonstrate the resolution + automation improvement in the handoff
  • Walk OJL supervisor through the apprentice-authored evidence map paired with the CBR (§9 of governing spec)
  • Supervisor scores the CBR; apprentice addresses any Beginning / Developing competencies per §8d (per-competency revision)
  • Capstone closes when every tagged competency rates Proficient or Advanced
