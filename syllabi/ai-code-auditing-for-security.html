Syllabus: AI Code Auditing for Security (Cyber Developer Associate)

Cyber Developer Associate (A-31, DRAFT 2026-05-08) — Phase 7: Advanced Security Integration Module 40 hours CURRENT

Course Description

AI-generated code is fast, fluent, and unverified. This hands-on course teaches you to audit it before it ships. You will read, critique, and remediate intentionally-flawed AI-generated code, configurations, and Dependency Injection setups — spotting hallucinations and slopsquatted dependencies, reviewing configurations for insecure defaults and exposed secrets, auditing dependency-injection wiring for vulnerabilities, confirming exploitability by tracing untrusted input to its sink, and remediating findings through a fix–verify–re-audit loop.

By the end of the course you can run a complete security audit of an AI-generated codebase and render a documented pre-deployment sign-off — practicing the discipline that nothing AI-authored ships until a human has audited it.

Learning Outcomes

  1. Explain the trust problem in AI-generated code — why plausibility is not correctness and fluency is not safety.
  2. Classify AI code defects into two failure families: hallucinations (things that don't exist) and vulnerabilities (things that are exploitable).
  3. Identify where and why AI code fails — training-data staleness, missing context, and confident wrong defaults.
  4. Describe the audit mandate: why AI-generated code is unverified until a deliberate security audit confirms it. (OJL 1.d)
  5. Describe the new attack surface AI-assisted development introduces: prompt → generated code → production pipeline.
  6. Explain the supply-chain and dependency risk that AI suggestions introduce, including hallucinated and typosquatted packages.
  7. Identify what a security auditor looks for in AI output that a functional reviewer misses. (OJL 1.d)
  8. Apply the threat model to predict where AI-generated changes most need auditing before deployment. (OJL 1.d)
  9. Describe a systematic, repeatable review process for AI-generated changes from intake to documented decision. (OJL 1.d)
  10. Apply audit scoping to determine what a pre-deployment review must cover for a given change. (OJL 1.d)
  11. Explain the strengths and blind spots of manual review versus tooling (linters, SAST, dependency scanners).
  12. Audit a small AI-generated diff end-to-end, producing a findings list with a verify-or-remediate decision for each. (OJL 1.d)
  13. Define an AI hallucination in code and audit a snippet to detect invented APIs, non-existent methods, and fabricated configuration keys. (OJL 1.d)
  14. Detect wrong library usage where the name is real but the signature or semantics are fabricated, and confirm the mismatch against the library's actual contract. (OJL 1.d)
  15. Explain why hallucinations are most dangerous when they almost work, and articulate the audit reflex that catches them before deployment.
  16. Verify AI-generated imports and symbols against authoritative documentation, distinguishing real-and-correct from real-but-misused and from outright invented. (OJL 1.d)
  17. Detect package hallucination and "slopsquatting" — AI-suggested packages that do not exist or that an attacker has registered under the hallucinated name — and audit a dependency list for them. (OJL 1.d)
  18. Confirm version and signature compatibility, auditing for APIs that exist in some version but not the one the project pins. (OJL 1.d)
  19. Confirm behavior, not just existence, completing the verification loop with a confirm-or-reject decision per dependency and API.
  20. Audit an AI-generated module end-to-end to detect hallucinated APIs, non-existent methods, and hallucinated/misused dependencies. (OJL 1.d)
  21. Confirm each suspected hallucination against an authoritative source — existence, version, signature, and behavior. (OJL 1.d)
  22. Produce a structured findings list with per-item evidence and a documented verify-or-reject decision suitable for a pre-deployment review. (OJL 1.d)
  23. Identify insecure defaults and exposed secrets/credentials in AI-generated configuration files, and explain why generators reach for them.
  24. Evaluate overly permissive settings — CORS, authentication, file/network permissions, and debug flags — against a least-privilege baseline. (OJL 1.d)
  25. Trace common environment and framework configuration mistakes AI tools make, mapping each to its concrete security impact. (OJL 1.d)
  26. Audit an AI-generated configuration set against a security checklist, documenting each finding with location and risk. (OJL 1.d)
  27. Explain what Dependency Injection is and identify how AI tools generate DI setups. (OJL 1.d)
  28. Identify the security-relevant moving parts of a DI setup — what gets wired, with what scope, and from where. (OJL 1.d)
  29. Evaluate why DI is a high-value audit target — central wiring and broad blast radius — relative to other code to audit. (OJL 1.d)
  30. Identify injection of untrusted or attacker-controllable beans/components in an AI-generated DI setup. (OJL 1.d)
  31. Evaluate scope and lifecycle defects — shared mutable singletons and leaked request state — and explain their security impact. (OJL 1.d)
  32. Audit for insecure or over-broad wiring and missing access boundaries between components. (OJL 1.d)
  33. Execute a systematic DI-audit pass — enumerate, trace provenance, check scope, verify boundaries — over a flawed setup. (OJL 1.d)
  34. Audit an AI-generated DI configuration end to end, applying the configuration checklist and the four-step DI-audit pass together. (OJL 1.d)
  35. Identify findings across the module's vulnerability classes — insecure config, untrusted injection, scope/lifecycle defects, over-broad wiring, and missing boundaries. (OJL 1.d)
  36. Document each finding with its location, risk, and recommended fix in an auditable findings report. (OJL 1.d)
  37. Identify the injection family — SQL, command, and path injection — as it appears in AI-generated code. (OJL 1.d)
  38. Identify broken authentication and access-control flaws in AI output, including missing checks and confused-deputy patterns. (OJL 1.d)
  39. Identify insecure deserialization, unsafe input handling, and sensitive-data exposure in AI-generated code. (OJL 1.d)
  40. Classify a flaw into the correct vulnerability class and explain why that class recurs in AI output.
  41. Trace a vulnerability class back to the prompt pattern that commonly elicits it.
  42. Trace untrusted input from its entry point to a dangerous sink to confirm a vulnerability is reachable. (OJL 1.d)
  43. Confirm exploitability by reasoning about whether a real input can drive the flaw, not just whether the pattern is present. (OJL 1.d)
  44. Separate real, exploitable findings from false positives and theoretical-only findings.
  45. Prioritize confirmed findings by severity into what blocks deployment versus what is logged.
  46. Identify the full set of vulnerability classes present in a complete AI-generated component. (OJL 1.d)
  47. Classify each finding into its vulnerability class with a precise location. (OJL 1.d)
  48. Trace untrusted input to its sink to confirm exploitability for each suspected finding. (OJL 1.d)
  49. Distinguish confirmed exploitable findings from false positives and theoretical-only items with documented reasoning. (OJL 1.d)
  50. Prioritize every confirmed finding by severity into block-vs-log dispositions.
  51. Execute the fix → verify → re-audit loop on a confirmed finding, treating remediation as a closed loop rather than a one-shot edit. (OJL 1.d)
  52. Remediate a security flaw without introducing a new one, applying minimal, targeted fixes and checking for regressions in adjacent code. (OJL 1.d)
  53. Verify that a fix actually closes the issue — re-testing the original exploit path and confirming the flaw no longer reproduces. (OJL 1.d)
  54. Decide when to remediate AI-authored code in place versus reject-and-regenerate, using flaw density, design soundness, and trust as the deciding factors.
  55. Define a pre-deployment audit gate for AI-generated code — its entry criteria, pass/fail conditions, and sign-off — that operationalizes the standard's "before deployment" clause. (OJL 1.d)
  56. Document an audit — findings, decisions, and evidence — so the gate decision is defensible under later review. (OJL 1.d)
  57. Integrate the audit gate into a team's review and CI/CD flow, choosing what is automated versus what requires a human auditor sign-off. (OJL 1.d)
  58. Audit a realistic AI-generated codebase end-to-end, integrating hallucination detection, configuration review, DI auditing, and vulnerability analysis into one coherent pass. (OJL 1.d)
  59. Confirm each finding's exploitability and prioritize it, separating real flaws from false positives across the whole codebase. (OJL 1.d)
  60. Remediate confirmed findings through the fix → verify → re-audit loop, or reject-and-regenerate where the code is too broken to patch. (OJL 1.d)
  61. Document the audit and render a pre-deployment sign-off decision — pass with attestation, or fail with the blocking findings named — backed by defensible evidence. (OJL 1.d)

Course Outline

1 Foundations of AI Code Auditing 6 hours
Lesson: 1: Why AI-Generated Code Must Be Audited (2h)
  • The trust problem: plausibility is not correctness, and fluency is not safety
  • Two failure families — hallucinations (things that don't exist) vs. vulnerabilities (things that are exploitable)
  • Where AI code fails: training-data staleness, missing context, confident wrong defaults
Lesson: 2: The Threat Model of AI-Assisted Development (2h)
  • The new attack surface: prompt → generated code → production pipeline
  • Supply-chain and dependency risk introduced by AI suggestions
  • What a security auditor looks for that a functional reviewer misses
Lesson: 3: The Code Audit Workflow (2h)
  • A systematic, repeatable review process for AI-generated changes
  • Scoping an audit: what a pre-deployment review must cover
  • Manual review vs. tooling (linters, SAST, dependency scanners) — strengths and blind spots
  • Activity: walk a small AI-generated diff through the audit workflow end-to-end
2 Spotting Hallucinations in AI-Generated Code 8 hours
Lesson: 4: What AI Hallucinations Look Like in Code (2h)
  • Invented APIs, non-existent methods, and fabricated configuration keys
  • Wrong library usage: right name, wrong signature, wrong semantics
  • Why hallucinations are most dangerous when they almost work
Lesson: 5: Detecting Hallucinated Dependencies and APIs (3h)
  • Verifying imports and symbols against authoritative documentation
  • Package hallucination and "slopsquatting": AI-suggested packages that don't exist (or that an attacker has registered)
  • Version and signature mismatches; confirming behavior, not just existence
  • Activity: verify a set of AI-suggested imports/APIs and flag the hallucinated ones
Lesson: 6: Hands-On — Hunting Hallucinations (3h)
  • Lab: audit an AI-generated module for hallucinated APIs, methods, and dependencies; produce a findings list with evidence and a verify-or-reject decision for each
3 Auditing Configurations and Dependency Injection 10 hours
Lesson: 7: Reviewing Configurations for Security Flaws (3h)
  • Insecure defaults, exposed secrets, and credentials in generated config
  • Overly permissive settings (CORS, auth, file/network permissions, debug flags)
  • Environment and framework config: what AI commonly gets wrong
  • Activity: audit an AI-generated configuration set against a security checklist
Lesson: 8: Understanding Dependency Injection and Its Risks (2h)
  • What Dependency Injection is and how AI tools generate DI setups
  • The security-relevant moving parts: what gets wired, with what scope, from where
  • Why DI is a high-value audit target — central wiring, broad blast radius
Lesson: 9: Auditing DI Setups for Vulnerabilities (3h)
  • Injection of untrusted or attacker-controllable beans/components
  • Scope and lifecycle defects (shared mutable singletons, leaked request state)
  • Insecure or over-broad wiring; missing access boundaries between components
  • A systematic DI-audit pass: enumerate, trace provenance, check scope, verify boundaries
Lesson: 10: Hands-On — Configuration and DI Audit Lab (2h)
  • Lab: audit an AI-generated DI configuration for the vulnerability classes above; document each finding with location, risk, and recommended fix
4 Identifying Security Flaws and Vulnerabilities 8 hours
Lesson: 11: Common Vulnerability Classes in AI-Generated Code (3h)
  • Injection (SQL/command/path), broken authentication and access control
  • Insecure deserialization, unsafe input handling, sensitive-data exposure
  • How these classes recur in AI output, and the prompt patterns that elicit them
Lesson: 12: From Recognition to Confirmation (2h)
  • Confirming exploitability: tracing untrusted input to a dangerous sink
  • Separating real vulnerabilities from false positives and theoretical-only findings
  • Severity and prioritization: what blocks deployment vs. what is logged
Lesson: 13: Hands-On — Full Vulnerability Audit (3h)
  • Lab: perform a complete vulnerability audit of an AI-generated component, confirming and prioritizing each finding with evidence of exploitability
5 Remediation and Pre-Deployment Review 8 hours
Lesson: 14: Security-Flaw Remediation Workflows (3h)
  • The fix → verify → re-audit loop; fixing the flaw without introducing a new one
  • Regression and re-test after remediation; confirming the fix actually closes the issue
  • When to remediate vs. reject-and-regenerate AI-authored code
  • Activity: remediate a confirmed finding and verify the fix
Lesson: 15: The Pre-Deployment Audit Gate (2h)
  • Gating AI-generated code before deployment: entry criteria and sign-off
  • Documenting the audit — findings, decisions, and evidence — for defensible review
  • Integrating the audit gate into a team's review and CI/CD flow
Lesson: 16: Capstone Audit — End to End (3h)
  • Integrative lab: audit a realistic AI-generated codebase from review → findings (hallucinations, config, DI, vulnerabilities) → confirmation → remediation → documented pre-deployment sign-off
  • Standard lesson (`type: lesson`) with an integrative audit activity — not a `type: capstone` summative lesson
