Syllabus: CI/CD Pipeline Concepts

Operations Support Specialist — Network 10 hours CURRENT

Course Description

CI/CD Pipeline Concepts introduces learners to the principles and practices of continuous integration and continuous deployment. This course covers version control strategies, automated build and testing processes, deployment methodologies, and popular CI/CD tools used in modern software development and operations environments. Learners will understand how CI/CD pipelines improve code quality, reduce deployment risk, and enable rapid feature delivery while maintaining system reliability.

Learning Outcomes

  1. Define continuous integration (CI) and continuous deployment (CD), and explain why teams adopt them.
  2. Distinguish trunk-based from feature-branch development workflows and identify when each fits.
  3. Configure GitHub branch protection rules with required status checks on a sample repository.
  4. Write CI-friendly commit messages using the conventional commits format.
  5. Identify the GitHub Actions events (`push`, `pull_request`, `schedule`) that trigger pipeline runs.
  6. Identify the four-level GitHub Actions object model: workflow, job, step, action.
  7. Locate and parse a workflow file in the `.github/workflows/` directory.
  8. Use marketplace actions (`actions/checkout`, `actions/setup-python`) to compose a workflow without writing custom shell.
  9. Author a one-stage workflow file that runs on every push and prints the Python version.
  10. Add a test job to an existing workflow that runs alongside the build job. (OJL 3b)
  11. Interpret status check results on a pull request and trace failures back to the failing step.
  12. Configure dependency caching with `actions/cache` to reduce workflow run time.
  13. Debug a red build by reading workflow logs and identifying root cause vs symptom.
  14. Distinguish jobs from steps and identify when work belongs in a separate job versus another step. (OJL 3b)
  15. Express job dependencies using `needs:` to build a directed pipeline graph.
  16. Use matrix strategies (`strategy.matrix`) to run a job across multiple language/OS versions in parallel.
  17. Identify use cases for reusable workflows and composite actions.
  18. Read a 6-stage / 2-environment promotion pipeline and explain how each stage gates the next.
  19. Distinguish repository secrets, organization secrets, and environment secrets, and identify when to use each. (OJL 3b)
  20. Configure a GitHub Environment with required reviewers and deployment branch restrictions.
  21. Reference environment-scoped secrets safely in workflow YAML using the `secrets.<NAME>` context.
  22. Describe how OIDC enables credential-less cloud deployment and why it's preferred over long-lived access keys.
  23. Read workflow run statuses from the GitHub UI and identify common failure patterns from the run summary.
  24. Add a status badge to a repository README using the GitHub Actions badge URL convention.
  25. Configure required status checks to enforce CI gating on the default branch.
  26. Identify the workflow-failure notification options available in GitHub and via third-party integrations.
  27. Apply pipeline observability practices: what to log, what to time, and where to surface results.
  28. Define blue-green deployment and identify the operational scenarios where its tradeoffs fit best. (OJL 3b)
  29. Describe canary releases and explain the role of the observation window in catching regressions early.
  30. Explain rolling updates and the role of the load balancer in coordinating node replacement.
  31. Compare deployment strategies for a given operational scenario and recommend one with rationale.
  32. Identify how each deployment strategy is expressed in a GitHub Actions workflow.
  33. Apply semantic versioning rules (MAJOR.MINOR.PATCH) to determine the correct version bump for a code change. (OJL 3b)
  34. Tag a release in Git and configure a tag-triggered GitHub Actions workflow.
  35. Generate release notes from conventional commit history.
  36. Execute a tag-based rollback when a deployment fails.
  37. Run post-deployment verification using smoke tests and health checks.
  38. Author a 4-stage CI/CD workflow (Lint → Test → Build → Deploy) on a course-provided sample repository. (OJL 3b)
  39. Configure a GitHub Environment with one protection rule and one consumed secret.
  40. Trigger the pipeline on pull request and produce a green end-to-end run.
  41. Document the pipeline shape in a README section that an external reader can use to understand the architecture.

Course Outline

1 CI/CD pipeline concepts and tooling 3 hours
Lesson: 1: CI/CD Fundamentals and Version Control for CI (1h)
  • What CI is, what CD is, and why teams adopt them
  • Trunk-based vs feature-branch development
  • Branch protection rules and required status checks
  • Conventional commits and CI-friendly history
  • Events that trigger pipelines (push, pull_request, schedule)
  • Individual: configure branch protection on a sample repo with one required status check
Lesson: 2: Introduction to GitHub Actions (1h)
  • Workflows, jobs, steps — the GitHub Actions object model
  • The `.github/workflows/` directory and YAML basics
  • Events, runners, and the marketplace actions ecosystem
  • Authoring a first one-stage workflow that builds a generic Python sample
  • Pair coding: write and commit a `build.yml` that runs on every push and prints the Python version
Lesson: 3: Build Automation and Automated Testing as a CI Step (1h)
  • Adding a test job alongside the build job (OJL 3b)
  • Status checks and the failing-build experience
  • Caching dependencies between runs (`actions/cache`)
  • Reading workflow logs and debugging a red build
  • Individual: extend the Lesson 2 workflow with a `pytest` job and intentionally break a test to observe the failure flow
2 Pipeline architecture 3 hours
Lesson: 4: Multi-Stage Pipelines (1h)
  • Jobs vs steps and when to split (OJL 3b)
  • `needs:` and the pipeline DAG
  • Matrix strategies for cross-version testing
  • Reusable workflows and composite actions
  • Worked example (instructor-led, not built by learners): a 6-stage / 2-environment promotion pipeline showing what production CI/CD architecture looks like
  • Pair coding: refactor a single-job workflow into two dependent jobs using `needs:`
Lesson: 5: Secrets, Environments, and Environment Protection (1h)
  • Repository secrets vs organization secrets vs environment-scoped secrets (OJL 3b)
  • Configuring a GitHub Environment with required reviewers
  • Deployment branches and tag-based gating
  • OIDC for cloud-credential-less deployment (overview)
  • Individual: configure a `staging` environment with one secret and one required reviewer
Lesson: 6: Monitoring and Feedback Loops (1h)
  • Workflow run statuses and surfacing them in the repo
  • Status badges in the README
  • Required checks on PRs and merge gating
  • Failure notifications: GitHub email, third-party (Slack/PagerDuty) overview
  • Pipeline observability — what to log, what to time
  • Individual: add a status badge to a repo README and configure required-check enforcement on the default branch
3 Automated testing and deployment 4 hours
Lesson: 7: Deployment Strategies (1.25h)
  • Blue-green deployments — what, when, the tradeoffs (OJL 3b)
  • Canary releases — gradual rollout, observation windows
  • Rolling updates — load-balancer-driven node replacement
  • How each strategy is expressed in a GitHub Actions workflow
  • ~20-minute in-lesson activity: short write-up choosing one strategy for a given scenario, with rationale
  • Individual write-up (in-lesson): "Given a small SaaS product with 2 production app servers, which deployment strategy would you recommend, and why?"
Lesson: 8: Release Management (1.25h)
  • Semantic versioning (MAJOR.MINOR.PATCH) and what bumps what (OJL 3b)
  • Tagging releases and the relationship between tags and pipelines
  • Auto-generated release notes
  • Rollback strategies — the previous-tag pattern, blue-green flip
  • Post-deployment verification — smoke tests, health checks
  • Pair coding: tag a release, observe the tag-triggered workflow run, then perform a tag-based rollback
