Learning Outcomes

  1. Describe what clean collaboration with an AI copilot means and how it differs from deferring to the tool.
  2. Explain why the developer retains full responsibility for AI-generated code regardless of who or what produced it.
  3. Classify copilot interactions as healthy collaboration versus unhealthy deference.
  4. Explain why small, scoped requests produce more reviewable output than whole-feature prompts.
  5. Apply prompting techniques that keep generated changes verifiable and within your mental model of the code.
  6. Identify when a request has grown too large to review and how to decompose it.
  7. Identify excessive code duplication as the canonical red flag in AI-generated code.
  8. Describe how boilerplate bloat, dead abstractions, and over-engineering show up in copilot output.
  9. Explain why plausible-looking code that does not fit the surrounding codebase is a warning sign.
  10. Define a "hallucination" in AI-generated code and identify common forms — invented APIs, non-existent methods, and wrong library usage.
  11. Identify insecure patterns a copilot commonly reaches for, such as a flawed dependency-injection setup or unsafe defaults. (OJL 1.d)
  12. Explain why fluent, confident output earns less trust, not more.
  13. Explain the blind-acceptance trap and why "it compiled" or "it looks right" is not verification.
  14. Apply a lightweight personal routine to verify AI-generated code before accepting it. (OJL 1.d)
  15. Identify when to reject and rewrite rather than patch AI-generated code.
  16. Describe what an AI-collaboration record captures: what was asked, what was accepted or rejected, and how it was verified. (OJL 6.c)
  17. Apply a lightweight, reusable format for documenting AI collaboration as you work. (OJL 6.c)
  18. Explain how transparent, traceable AI-use documentation supports peer code review.
  19. Apply your AI-collaboration record to present an AI-assisted change in a code review or technical assessment. (OJL 6.c)
  20. Explain how to answer "did you understand this code?" by demonstrating ownership rather than deflecting.
  21. Apply an end-to-end walkthrough that defends an AI-assisted change from prompt to verified commit. (OJL 6.c)

Course Outline

1 Collaborating Cleanly with AI Copilots 3 hours
Lesson: 1: The Copilot as Collaborator, Not Author (1.5h)
  • What "clean collaboration" means: directing the copilot vs. deferring to it
  • The copilot as a fast junior pair — useful, fallible, and never the decision-maker
  • Where responsibility stays with the developer, no matter who typed the code
Lesson: 2: Driving the Copilot — Scoped, Verifiable Requests (1.5h)
  • Small, scoped prompts vs. "generate the whole feature" dumps
  • Keeping generated changes reviewable and asking for explanations you can check
  • Maintaining your own mental model so you never ship code you cannot read
2 Recognizing Red Flags in AI-Generated Code 4 hours
Lesson: 3: Red Flags — Duplication, Bloat, and Copy-Paste Sprawl (1.5h)
  • Excessive code duplication as the canonical red flag of AI output
  • Boilerplate bloat, dead abstractions, and over-engineered suggestions
  • Plausible-looking code that does not fit the surrounding codebase
Lesson: 4: Hallucinations and Insecure Patterns (1.5h)
  • Hallucinations: invented APIs, non-existent methods, and wrong library usage
  • Recognizing insecure patterns the copilot reaches for — unsafe defaults, a flawed dependency-injection setup — at the level needed to reject them
  • Why fluent, confident output earns less trust, not more
Lesson: 5: Avoiding Blind Acceptance — Verify Before You Commit (1h)
  • The blind-acceptance trap: "it compiled" and "it looks right" are not verification
  • A lightweight personal routine to verify AI-generated code before accepting it
  • Knowing when to reject and rewrite rather than patch
3 Documenting AI Collaboration for Assessment 3 hours
Lesson: 6: Writing the AI-Collaboration Record (1.5h)
  • What to capture: what you asked, what you accepted or rejected, and how you verified it
  • Making AI use transparent and traceable rather than hidden
  • A reusable, lightweight format for documenting collaboration as you work
Lesson: 7: Passing Technical Assessments and Peer Code Reviews (1.5h)
  • Presenting your AI-collaboration approach in a code review or technical assessment
  • Answering "did you understand this code?" — demonstrating ownership, not deflection
  • Walkthrough: defend an AI-assisted change end-to-end, from prompt to verified commit
