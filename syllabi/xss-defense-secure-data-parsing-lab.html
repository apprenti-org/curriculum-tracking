Syllabus: XSS Defense & Secure Data Parsing Lab

Cyber Developer Associate — Phase 6 (Secure Web and Front-End Development) 24 hours CURRENT

Course Description

Course description pending.

Learning Outcomes

  1. Trace untrusted input from an HTTP request through application code to the exact DOM sink where it is rendered, and explain why that path is the unit of analysis for Cross-Site Scripting (XSS) defense.
  2. Reproduce a reflected XSS attack by submitting a script payload through a query parameter and confirming it executes in the browser.
  3. Identify the common dangerous DOM sinks — `innerHTML`, `outerHTML`, `insertAdjacentHTML`, `document.write`, and inline event-handler assignment — that turn untrusted strings into executing markup.
  4. Neutralize a reflected XSS by replacing the dangerous sink with `textContent` (or another safe DOM API) so the payload renders as inert literal text.
  5. Build a reusable sink map that catalogs each dangerous sink, its safe counterpart, and the rendering context in which it appears.
  6. Explain how stored Cross-Site Scripting (XSS) differs from reflected XSS, and why persisted attacker-controlled data executes for every viewer on render.
  7. Reproduce a stored XSS by persisting a script payload through an application's data store and confirming it executes when the stored data is later rendered to the DOM.
  8. Refactor a render routine that builds list or feed markup with `innerHTML` so that persisted data is rendered with `textContent` and safe DOM-construction APIs instead.
  9. Implement encode-on-output discipline so that the same persisted value is rendered safely at every DOM update, not just the first.
  10. Identify DOM-based Cross-Site Scripting (XSS) as a client-side-only vulnerability whose source and sink both live in JavaScript and never reach the server.
  11. Trace a DOM-based XSS from a client-side source such as `location.hash` to a dangerous sink, and explain why server-side defenses cannot see it.
  12. Refactor `innerHTML` and `insertAdjacentHTML` sinks to `textContent` and safe DOM APIs to neutralize a DOM-based XSS.
  13. Determine when `createElement` plus `append` is the correct choice over string-built HTML, and apply it to build dynamic content safely.
  14. Reproduce a `javascript:` URL Cross-Site Scripting (XSS) by assigning an untrusted value to an `href` or `src`, and neutralize it by validating the URL scheme before assignment.
  15. Identify attribute-injection sinks where untrusted data is written into element attributes, and use `setAttribute` safely while avoiding dangerous-attribute assignment.
  16. Reproduce an event-handler XSS through inline `on` attribute assignment and replace it with `addEventListener` so untrusted data never becomes executable code.
  17. Verify that each context-specific fix blocks its payload, and extend the sink map with rows for URL, attribute, and event-handler contexts.
  18. Explain when rendering untrusted HTML is a genuine requirement and why `textContent` cannot be used in that case, framing sanitization as a Cross-Site Scripting (XSS) output defense of last resort.
  19. Implement safe rendering of untrusted HTML with a vetted sanitizer (DOMPurify), configuring it to strip script, event-handler, and dangerous-URL vectors before assignment to the DOM.
  20. Implement a manual allow-list sanitizer that parses untrusted HTML and keeps only permitted tags and attributes, and explain why allow-listing beats blocklisting.
  21. Verify that sanitized output actually neutralizes the payload by re-running attacks against the rendered result and confirming no script executes.
  22. Explain why every value that crosses into the browser — including a `fetch` response body — is untrusted input at the moment `JSON.parse` runs, and why "never assume shape" is the foundational client-side parsing rule.
  23. Parse untrusted JSON safely by wrapping `JSON.parse` so that a malformed or truncated payload is caught and converted to a safe rejection rather than an unhandled exception.
  24. Validate parsed JSON against an explicit shape — confirming required keys, types, and structure — before any value is read or used downstream.
  25. Implement type and bounds checks on individual parsed values as the final gate before the data is handed to application logic.
  26. Explain how the JavaScript prototype chain makes `__proto__`, `constructor`, and `prototype` keys dangerous when they arrive inside parsed JSON.
  27. Reproduce a prototype-pollution attack in which a crafted JSON payload flowing through a naive merge or property-assignment routine corrupts `Object.prototype`.
  28. Defend object construction against pollution using null-prototype objects, explicit key guards, and `Object.hasOwn` for property reads.
  29. Implement a safe deep-merge that rejects dangerous keys and preserves the defense across nested input.
  30. Detect accidental data leakage caused by over-broad `JSON.stringify` that serializes more of an object than was intended.
  31. Serialize objects safely by controlling the emitted output with a `replacer` allow-list and `toJSON` methods.
  32. Defend against `JSON.stringify` runtime hazards — circular references, value types that serialize unexpectedly, and silently dropped values.
  33. Implement a controlled round-trip that pairs a safe `replacer` on serialize with a guarded `reviver` on parse.
  34. Detect unsafe dynamic property access where a key derived from untrusted input is used to read or write an object.
  35. Defend against client-side mass assignment by copying only an allow-list of fields from a parsed object onto an application object.
  36. Identify type-coercion and loose-equality bugs that let untrusted values bypass validation checks.
  37. Implement safe object-handling patterns that combine key guards, allow-list copying, and strict comparison for untrusted data.
  38. Assemble a three-stage client-side pipeline — fetch, parse, render — and identify the distinct trust boundary and control point that sits between each stage.
  39. Apply defensive parsing to a fetched response, validating shape and types and guarding against prototype-pollution keys before any parsed value is held as application data.
  40. Harden the render stage so that every untrusted value reaches the Document Object Model (DOM) through a safe sink (`textContent`, safe DOM APIs, or a vetted sanitizer) and never through a dangerous one.
  41. Verify the assembled pipeline end to end, demonstrating that no untrusted value reaches a dangerous sink and that a malformed or hostile response fails closed without crashing the page.
  42. Audit a small client-side application to locate Cross-Site Scripting (XSS) and insecure-parsing flaws by tracing untrusted data from its source to its sink.
  43. Prioritize the findings by exploitability and impact, distinguishing a directly exploitable sink from a latent or lower-risk weakness.
  44. Remediate each finding by replacing the dangerous pattern with the correct safe one — safe DOM sinks for rendering flaws, defensive parsing and prototype-pollution guards for data flaws.
  45. Confirm each fix by re-running the attack payload that previously succeeded and demonstrating it no longer executes or corrupts application state.
  46. Explain defense in depth for a client-side feature and identify the independent layers that must each stop an attack on its own.
  47. Harden a realistic feature that combines rendering and data parsing by layering safe DOM manipulation, sanitization, schema validation, and safe object handling.
  48. Verify layer independence by disabling one control and demonstrating that a downstream layer still blocks the attack, then restoring the control.
  49. Demonstrate that the hardened feature resists a scripted attack set, producing evidence that every payload is neutralized and application state stays intact.

Course Outline

1 XSS Defense Labs 10 hours
Lesson: 1: Reflected XSS and the Sink Map (2h)
  • Trace untrusted input from request to DOM sink
  • Reproduce a reflected XSS, then neutralize it with `textContent` and contextual output handling
  • Build a reusable map of dangerous DOM sinks to watch for
Lesson: 2: Stored XSS and Safe Rendering (2h)
  • How persisted, attacker-controlled data executes on render
  • Render stored data without executing it
  • Encode-on-output discipline across DOM updates
Lesson: 3: DOM-Based XSS — innerHTML vs. textContent (2h)
  • Identify client-side-only XSS that never reaches the server
  • Refactor `innerHTML` / `insertAdjacentHTML` sinks to `textContent` and safe DOM APIs
  • When `createElement` + `append` beats string-built HTML
Lesson: 4: Dangerous Contexts — URLs, Attributes, and Event Handlers (2h)
  • `javascript:` URLs and unsafe `href` / `src` assignment
  • Attribute injection and `setAttribute` pitfalls
  • Inline event handlers (`on`) as XSS sinks and their safe alternatives
Lesson: 5: Rendering Untrusted HTML Safely (2h)
  • When HTML must be rendered: sanitization as XSS defense
  • Using a vetted sanitizer (DOMPurify) and a manual allow-list approach
  • Verifying the sanitized output actually blocks the payload
2 Secure Data Parsing Labs 8 hours
Lesson: 6: Defensive JSON.parse on Untrusted Input (2h)
  • Never trust the shape: validate parsed JSON against an expected schema
  • Handle malformed, partial, and hostile payloads without crashing
  • Type and bounds checks before using parsed values
Lesson: 7: Prototype Pollution via Insecure Object Handling (2h)
  • How `__proto__` / `constructor` keys poison objects during merge or assign
  • Reproduce a prototype-pollution attack sourced from parsed JSON
  • Defend with null-prototype objects, key guards, and safe merge
Lesson: 8: JSON.stringify Pitfalls and Safe Serialization (2h)
  • Accidental data leakage through over-broad serialization
  • `replacer` functions and `toJSON` to control what is emitted
  • `reviver` functions on parse; circular-reference and type hazards
Lesson: 9: Insecure Object Handling Patterns (2h)
  • Unsafe dynamic property access and client-side mass assignment
  • Type coercion and equality bugs that bypass checks
  • Safe object-handling patterns for untrusted data
3 Integrated Client-Side Defense Labs 6 hours
Lesson: 10: Secure Render Pipeline — fetch → parse → render (2h)
  • Consume an API, parse the response defensively, and render to the DOM safely
  • Place each control at the right layer of the pipeline
  • Verify no untrusted value reaches a dangerous sink
Lesson: 11: Find-and-Fix Audit Lab (2h)
  • Audit a small vulnerable client app for XSS and unsafe-parsing flaws
  • Prioritize and remediate each finding
  • Confirm fixes with the attack payloads that previously succeeded
Lesson: 12: Defense-in-Depth Scenario Lab (2h)
  • Harden a realistic feature that combines rendering and data parsing
  • Layer safe DOM APIs, sanitization, schema validation, and safe object handling
  • Demonstrate the feature resists a scripted set of attacks
